## Priority 1: Cryptographic Hash (Quick Win)

**What:** Replace DJB2 with SHA-256 or BLAKE3.

**Why:** DJB2 produces 32-bit hashes with easy collisions. SHA-256 produces 256-bit hashes that are collision-resistant, preimage-resistant, and the standard for content addressing.

**Where:** Two functions in `SpellCeremony.tsx`:

```typescript
// CURRENT (DJB2 — not cryptographic)
function hashConstellation(nodes: CircuitNode[]): string {
  const str = nodes.map(n => n.id).join(':');
  let hash = 0;
  for (let i = 0; i < str.length; i++) {
    const char = str.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash;
  }
  return Math.abs(hash).toString(36);
}

// UPGRADE (SHA-256 via Web Crypto API — available in all browsers)
async function hashConstellation(nodes: CircuitNode[]): Promise<string> {
  const str = nodes.map(n => n.id).join(':');
  const data = new TextEncoder().encode(str);
  const hashBuffer = await crypto.subtle.digest('SHA-256', data);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('').slice(0, 12);
  // First 12 hex chars = 48 bits, readable but collision-resistant
}
```

**Impact:** Same blade, same constellation, same hash — but now cryptographically meaningful. Existing blade hashes would change (breaking backward compatibility with exported blades), so consider adding a `version` field to the proof.

**Effort:** ~30 minutes. One function swap + make the proof generation async.

## Priority 2: Signature with Key Pair

**What:** Sign each blade with a locally-generated private key. The signature proves WHO forged it.

**Why:** Currently `SPELL-7IRRH6-8` could be generated by anyone who knows the inputs. With a key-signed signature, only the holder of the private key could have produced it.

**How:**

```typescript
// Generate keypair on first visit (stored in localStorage or IndexedDB)
const keyPair = await crypto.subtle.generateKey(
  { name: 'ECDSA', namedCurve: 'P-256' },
  false, // not extractable
  ['sign', 'verify']
);

// Sign the blade proof
async function signBlade(proof: SpellProof, privateKey: CryptoKey): Promise<string> {
  const data = new TextEncoder().encode(
    `${proof.constellationHash}:${proof.lapCount}:${proof.duration}:${proof.bladeHex}`
  );
  const signature = await crypto.subtle.sign(
    { name: 'ECDSA', hash: 'SHA-256' },
    privateKey,
    data
  );
  return btoa(String.fromCharCode(...new Uint8Array(signature)));
}
```

**The inception key pattern:** The first keypair generated IS the inception key from Act XXVII. It signs the first blade. Subsequent blades are signed by the same key, creating an identity chain. The key retires when the user exports it to a hardware wallet or DID.

**Privacy note:** The key stays local. Never sent to any server. The public key can be shared when the user chooses to enter the trust graph. Until then, the blades are signed but private — ZK in spirit.

**Effort:** ~2 hours. Key generation + storage + sign function + export public key option.

## Priority 3: Hash Chain (Blade Lineage)

**What:** Each blade references the hash of the previous blade forged by the same key. Creates a tamper-evident chain.

**Why:** Currently blades are isolated artifacts. With a hash chain, they form a lineage — a verifiable sequence of ceremonies. Tampering with any blade breaks the chain from that point forward.

**How:**

```typescript
interface BladeProof {
  // ... existing fields
  previousBladeHash: string | null; // null for inception blade
  bladeHash: string; // SHA-256 of this blade's canonical data
  chainLength: number; // position in the chain
}

function computeBladeHash(proof: BladeProof): string {
  const canonical = JSON.stringify({
    constellation: proof.constellationHash,
    laps: proof.lapCount,
    duration: proof.duration,
    hex: proof.bladeHex,
    stratum: proof.bladeStratum,
    previous: proof.previousBladeHash,
    timestamp: proof.timestamp
  });
  return sha256(canonical);
}
```

**The chain tells the story:** Universe → Sun → Selene → [next blade] → ... Each blade carries the weight of every blade before it. The chain is the trust graph in miniature — one forger's accumulated practice.

**Effort:** ~1 hour. Add previousBladeHash to proof, compute on forge, store in inventory.

## Priority 4: Commitment Scheme (Pre-Evocation Lock)

**What:** Commit to the constellation hash BEFORE evocation starts. Reveal after forging. Proves the constellation wasn't modified mid-ceremony.

**Why:** Currently you could theoretically modify the constellation during evocation and the proof wouldn't know. A commitment scheme separates the "I chose this path" moment from the "I walked this path" moment.

**How:**

**Architecturally:** The commitment is the Swordsman's act — locking the boundary before the Mage projects through it. "We cannot merge" applied to the proof system. The constellation is fixed before the ceremony begins.

**Effort:** ~1 hour. Commitment on evoke start, verification on evoke end, include both in proof.

## Priority 5: Timestamp Anchoring (External Witness)

**What:** Publish blade hashes to an external timestamping service. The blade proves WHEN it was forged, verified by a third party.

**Why:** Currently the timestamp is self-reported from the browser. An external anchor makes it independently verifiable.

**Options (least to most sovereign):**

**Option A: Zcash shielded memo.** Publish the blade hash as a memo in a shielded transaction. The Zcash blockchain timestamps it. The memo is private (shielded) but the timestamp is public. Architecturally perfect — the privacy chain timestamps the privacy proof.

**Option B: IPFS/Pinata.** Pin the blade proof as a JSON file. The CID is content-addressed. Not timestamped by a chain but immutably stored.

**Option C: OP_RETURN on Bitcoin/Ethereum.** Cheapest option for timestamp anchoring. Public but minimal data (just the hash).

**Option D: Ceramic/ComposeDB.** Decentralised data network with built-in timestamping. Closer to the VRC/trust graph architecture.

**Recommendation:** Start with Zcash (Option A) — it aligns with Act IX, the shielded pool narrative, and the privacy-first architecture. The blade hash in a shielded memo is the forge's proof-of-existence anchored to the privacy chain.

**Effort:** Variable. Zcash integration ~4-8 hours (needs a wallet/signing flow). IPFS ~1-2 hours.

## Priority 7: ZK Circuit (The Real Thing)

**What:** Implement an actual zero-knowledge proof circuit for blade claims.

**Why:** The architecture claims "the blade proves without revealing." Currently the blade reveals its full constellation path. A ZK circuit would let you prove "I forged a blade with stratum ≥ 5 on a constellation containing con-gap" without revealing the full path.

**How:** This is the hardest step. Options:

**Circom/SnarkJS:** Mature ZK tooling. Write a circuit that takes the constellation path as private input, computes the hash and stratum as public output. The proof verifies that the forger knows a path that produces the claimed hash and stratum, without revealing the path.

**Noir (Aztec):** Rust-like ZK DSL. Easier to write than Circom. Browser-compatible via WASM.

**What to prove:**
- "I know a constellation path that hashes to X" (preimage proof)
- "My blade has stratum ≥ N" (range proof on Hamming weight)
- "My constellation includes node Y" (membership proof)
- "I forged ≥ K laps" (range proof on traversal depth)

**Architecturally:** This is the moment the forge's name becomes literally true. The ZK Swordsman Blade Forge stops being a narrative description and becomes a cryptographic implementation.

**Effort:** Significant. 2-4 weeks for a basic circuit. But the architecture is already specified — the six dimensions, the lattice, the hex encoding all map directly onto circuit constraints.
